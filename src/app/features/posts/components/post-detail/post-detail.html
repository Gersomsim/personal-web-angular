El Problema

Era un viernes por la tarde cuando recibimos la alerta: la página de reportes tardaba más de 10 segundos en cargar . Los usuarios empezaban a quejarse y el equipo de soporte estaba saturado.

El sistema llevaba meses funcionando sin problemas, pero el crecimiento de datos había superado las expectativas. Lo que antes eran 10,000 registros ahora eran más de 500,000.

Advertencia

Nunca optimices prematuramente. Primero mide, luego optimiza. En nuestro caso, el problema solo apareció cuando alcanzamos escala real.

Diagnóstico

Antes de tocar código, necesitaba entender qué estaba pasando. Usé varias herramientas para diagnosticar el problema:

  • PostgreSQL EXPLAIN ANALYZE para ver el plan de ejecución
  • pg_stat_statements para identificar las consultas más lentas
  • Application Performance Monitoring para ver el tiempo total de respuesta

Identificando el cuello de botella

La consulta problemática era un JOIN entre tres tablas con filtros dinámicos. Aquí está la versión simplificada:

sql 
SELECT o.id, o.total, c.name, p.title
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN products p ON o.product_id = p.id
WHERE o.created_at BETWEEN $1 AND $2
  AND c.region = $3
ORDER BY o.created_at DESC
LIMIT 100;

El EXPLAIN ANALYZE reveló que PostgreSQL estaba haciendo un Sequential Scan en la tabla de orders, revisando cada una de las 500,000 filas.

La Solución

La solución involucró dos cambios principales: añadir índices correctos y refactorizar la consulta para aprovechar mejor el query planner de PostgreSQL.

Añadiendo índices correctos

Pro tip

Un índice compuesto es más eficiente que múltiples índices simples cuando siempre filtras por las mismas columnas juntas.

Creé un índice compuesto que cubría los filtros más comunes:

sql 
CREATE INDEX CONCURRENTLY idx_orders_date_customer
ON orders (created_at DESC, customer_id)
WHERE deleted_at IS NULL;

Refactorizando la consulta

También reestructuré la consulta para usar un CTE (Common Table Expression) que primero filtra los orders antes de hacer los JOINs:

sql 
WITH filtered_orders AS (
  SELECT id, total, customer_id, product_id
  FROM orders
  WHERE created_at BETWEEN $1 AND $2
    AND deleted_at IS NULL
  ORDER BY created_at DESC
  LIMIT 100
)
SELECT fo.id, fo.total, c.name, p.title
FROM filtered_orders fo
JOIN customers c ON fo.customer_id = c.id
  AND c.region = $3
JOIN products p ON fo.product_id = p.id;

El cambio se desplegó sin downtime gracias a CREATE INDEX CONCURRENTLY . Los usuarios notaron la mejora inmediatamente.

Conclusiones

Este caso me dejó varias lecciones importantes:

  1. Mide antes de optimizar — Sin EXPLAIN ANALYZE, habría adivinado mal el problema.
  2. Los índices no son magia — Un índice mal diseñado puede empeorar las cosas.
  3. La estructura de la consulta importa — A veces reescribir es mejor que indexar.
  4. Monitorea proactivamente — Este problema se habría detectado antes con alertas de performance.
"La optimización prematura es la raíz de todos los males, pero la optimización tardía es la raíz de todos los incidentes de producción."
